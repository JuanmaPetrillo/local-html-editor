# Security Model

## Core principle

Imported HTML is untrusted.

A presentation generated by Copilot can include arbitrary HTML, CSS, JavaScript, inline event handlers, SVG, remote assets, tracking pixels, form submissions, and browser APIs. The editor must not assume that imported content is safe.

## Required security properties

The app must ensure:

- imported content cannot access filesystem APIs
- imported content cannot call privileged app commands
- imported content cannot exfiltrate data over the network in safe mode
- imported content cannot navigate or replace the trusted app shell
- imported content cannot silently load remote scripts, images, fonts, or beacons in safe mode
- app logs do not capture sensitive content unnecessarily
- exported HTML does not leak absolute local paths

## Modes

### Safe design mode

Default. Required behavior:

- scripts disabled
- forms disabled or inert
- remote connections blocked
- app bridge unavailable
- preview runs in a sandboxed iframe/webview
- editable actions are controlled by the app, not by imported JS

### Interactive preview mode

Optional. Required behavior:

- explicit user action to enter
- separate isolated webview/session
- no app bridge
- no filesystem scope
- remote network blocked by default
- clear visual label: `Interactive Preview`
- easy return to safe design mode

## Network policy

Default blocked:

- `fetch`
- XHR
- WebSocket
- EventSource
- `sendBeacon`
- remote scripts
- remote CSS
- remote images
- remote fonts
- external navigation

Allowed by default:

- local app preview protocol
- `data:` for inlined assets where safe
- `blob:` for generated local preview objects where safe

## Filesystem policy

- File access must come from explicit user selection or narrow app-scoped storage.
- Do not grant preview content filesystem permissions.
- Do not store project files outside user-approved locations unless app storage is clearly documented.
- Do not write over the original import unless the user explicitly chooses overwrite.

## Sanitization policy

DOMPurify or similar sanitization may be used for fragments inserted into the trusted app shell.

Do not destructively sanitize the entire imported presentation as the only preservation strategy, because it may break legitimate layout and interactions. Instead:

- classify content
- preserve original source
- isolate preview
- expose only safe editable operations

## Security acceptance criteria

A malicious fixture must not be able to:

- call app APIs
- read local files
- write local files
- make remote requests in safe mode
- navigate the app shell
- escape the preview container
- weaken CSP or sandbox settings
